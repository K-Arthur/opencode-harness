1. Detailed Feature Breakdown & Specifications
Feature 1: Agent-Aware Chat Interface (The Core): This is the primary interaction point, designed to be superior to a terminal.

Display conversation history with timestamps and speaker roles (user, assistant, system).

Skill Usage Cards: When the agent activates a skill, the view must clearly show which skill was loaded, for how long, and any output from its activation.

Tool Call Log: Display a real-time timeline of every tool call the agent makes (e.g., read_file, edit_file, run_terminal_command). Each tool call should be expandable to show its input arguments (e.g., the file path it's reading) and the final result. This solves the "black box" problem in the terminal.

Inline Code Blocks & Diffs: For code suggestions, present a unified diff view so the user can exactly see what lines will be added, changed, or deleted before accepting.

Feature 2: Intelligent Context Engine: This is the feature for delivering a 10x boost. It must automatically curate a high-signal context package for the agent.

Core Context: Always include the content of all open editor tabs.

Structured Context: Include the current file's Abstract Syntax Tree (AST) structure (via a language server) to give the AI better code understanding.

Diagnostics Context: Automatically send opencode the current file's linter errors (ESLint, TypeScript compiler diagnostics) along with the code.

Workspace Awareness: Include the project's directory tree (limited depth), relevant package.json/pyproject.toml/Cargo.toml files, active git branch, and recent git diff.

Terminal Capture: Allow the user to select a block from the VS Code terminal and send it as context.

Feature 3: Keyboard-Driven Workflow: High productivity requires keyboard control. All major features must have customizable keybindings. Open the chat, trigger an inline action, execute a rollback—all without touching the mouse.

2. Technical Architecture & Implementation Plan
The core design is based on a "Client-Server" model. opencode is the existing server, and the VS Code extension is the powerful, purpose-built client.

Provider-Agnostic Core: The extension will not have a direct API key for LLMs. Instead, it will always communicate by spawning or connecting to the opencode CLI process via Node.js's child_process module. This leverages opencode's existing provider support (OpenAI, Anthropic, Local models) and keeps the extension simple and maintainable.

The SessionManager: A core module that manages the lifecycle of an opencode process. It will:

Spawn the process and keep it alive.

Handle all stdin/stdout communication.

Parse opencode's output to separate chat text from structured data (like skill usage or tool calls).

Expose an event-driven API for the rest of the extension.

Diff Generation Engine on Acceptance: Instead of directly writing AI code to a file, the extension will temporarily hold the suggested changes. When the user clicks "Accept," a diff generator will compare the AI's output with the original file and use the VS Code API (workspace.applyEdit) to create a clean, undoable edit. This allows for accepting only parts of a suggestion and provides native VS Code history.

Performance: All communication with opencode and any intensive processing (like diff generation) must happen in a Node.js worker thread or a separate process to prevent the UI from freezing.

3. User Experience (UX) & Interface Guidelines
Layout Composition:

Primary Sidebar (Activity Bar): Custom view for "OpenCode Harness," housing the main chat and history.

Inline (Editor Context Menu & CodeLenses): For context-specific actions.

Status Bar: For the "Context Usage" ring and a simple indicator showing if opencode is idle or thinking.

Quick Pick Menu (Command Palette): Accessible via keyboard shortcuts for all major commands.

4. Non-functional Requirements
Performance:

The extension should not increase VS Code's startup time by more than 50ms.

UI interactions (like typing in the chat) must remain responsive even while the AI is generating a reply.

Context window estimation should cost less than 10ms.

Reliability:

The extension must gracefully recover if the opencode process crashes, providing an option to restart the session without losing chat history.

All file operations should be transactional to prevent data corruption.

Platform Support (Arch & RPM): The extension must be packaged as a standard .vsix and available on the VS Code Marketplace, ensuring compatibility with all VS Code versions found on recent Arch Linux and Fedora/RHEL releases.

5. Development Roadmap (MVP)
Phase 1: Foundation & Core Client (Week 1-2): Project scaffolding, the SessionManager to spawn/control opencode, and a basic read-only chat output channel.

Phase 2: Interactive Features (Week 3-4): Implement the rich chat view with clear tool calls and skill usage cards. Add inline code actions and diff-based code insertion.

Phase 3: Context & Control (Week 5-6): Implement the context engine (open files, linter errors). Integrate keyboard shortcuts and the status bar indicator.

Phase 4: Advanced & Polish (Week 7-8): Add Context Monitor, Skill Manager UI, checkpoint/rollback system, and full platform-specific testing on Arch and Fedora.

6. Required Development Environment Setup (for Arch/RPM)
To start building, a developer will need to set up their environment as follows:

Install Node.js: Ensure Node.js version 20.x or later is installed. On Arch: sudo pacman -S nodejs npm. On Fedora: sudo dnf install nodejs.

Install OpenCode: Follow the official instructions for your distribution. This will likely involve installing it via npm or from a release binary.

Install VS Code Extension Dependencies: The libsecret library is required for certain credential-storing modules used in the build/publish toolchain.

Arch: sudo pacman -S libsecret

Fedora/RHEL: sudo yum install libsecret-devel

VS Code Yeoman Generator: Use the official generator to scaffold the project.

npm install -g yo generator-code

Then, yo code and select New Extension (TypeScript).

VS Code Launch Configuration: Set up a .vscode/launch.json file for debugging. The standard configuration will launch a new VS Code window (the "Extension Development Host") with the extension loaded. Pressing F5 will start this debug session, allowing you to set breakpoints directly in the extension code.
