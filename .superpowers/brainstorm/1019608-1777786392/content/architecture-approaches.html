Pros

  • Simple: single process, no port management
  • Used by Zed, JetBrains (proven pattern)
  • No authentication needed (local stdio)

Cons

  • Limited to ACP's message protocol
  • No file search, diff, session management APIs
  • Harder to stream real-time events
  • Missing /undo, /redo commands per ACP docs

Pros

  • Full API: sessions, files, diffs, search, events
  • SSE streaming for real-time agent visibility
  • Type-safe SDK (@opencode-ai/sdk)
  • Session management + diff endpoint built in
  • Same protocol the TUI uses internally

Cons

  • Must manage server process lifecycle
  • Port management required
  • Slightly more setup code

Verdict

Approach B (HTTP Server + SDK) is the clear winner. The ACP protocol is designed for generic editor integration but the HTTP server exposes the exact APIs our extension needs: session diffs, file search, event streaming, and full session management. The SSE event stream directly enables the "agent visibility" feature (showing what tool is being called, which skill is loaded, etc.).
